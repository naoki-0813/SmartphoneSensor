## コミュニケーション方針

- **やり取りはすべて日本語**で行う。説明・要約・質問も日本語。
- 利用者は IT 初心者の可能性がある。**専門用語には短い補足**を添える
  （例: 「ビルド（= 公開用にコードをまとめる処理）」）。
- 大きな変更や、元に戻しにくい操作（削除・push・外部送信）の前には、
  **一言確認**してから進める。
- わからないこと・前提が曖昧なことは、勝手に決めずに**質問する**。

## よく使うコマンド

パッケージマネージャは **pnpm** を使う（npm/yarn は使わない）。

| やりたいこと | コマンド |
| --- | --- |
| 依存をインストール | `pnpm install` |
| 開発サーバ起動 | `pnpm dev` |
| 本番ビルド | `pnpm build` |
| Lint（書き方チェック） | `pnpm lint` |
| 自動整形 | `pnpm format` |
| テスト実行 | `pnpm test` |

## コーディング規約

### TypeScript
- `strict` 前提。`any` は原則使わない（どうしても必要なら理由をコメント）。
- 型は明示しすぎず、推論できるところは任せる。公開関数の引数・戻り値は型を書く。
- `null`/`undefined` は早期 return で素直に処理する。

### React
- **関数コンポーネント + フック**のみ。クラスコンポーネントは使わない。
- コンポーネントは **named export**（`export function Button() {}`）。
- 1 ファイル 1 コンポーネントを基本とし、ファイル名は `PascalCase.tsx`。
- props は型（`type XxxProps`）で定義する。
- 状態は必要最小限。サーバ状態とUI状態を混ぜない。

### 命名
- 変数・関数: `camelCase` / コンポーネント・型: `PascalCase` / 定数: `UPPER_SNAKE_CASE`
- 真偽値は `is`/`has`/`can` で始める（例: `isLoading`）。
- ハンドラは `handle` + 対象（例: `handleSubmit`）。

### ディレクトリ構成（目安）
```
src/
├─ components/   再利用するUI部品
├─ pages/        画面単位
├─ hooks/        カスタムフック
├─ lib/          ロジック・ユーティリティ
└─ types/        共通の型
```

### コメント
- **何を**ではなく**なぜ**を書く。周囲のコードのコメント密度に合わせる。
- コメントも日本語で良い。

## 作業の進め方

1. まず関連ファイルを読み、既存のやり方に合わせる（独自流を持ち込まない）。
2. 変更は**小さく**。1 つの目的＝1 つのまとまり。
3. 変更後は可能なら `pnpm lint` と `pnpm test` を通す。失敗したら正直に報告する。
4. 動作確認が必要なものは `/run` や `/verify` を使って実際に確かめる。

## Git / コミット規約

- コミットメッセージは**日本語**。1 行目に種別をつける（Conventional Commits 風）。
  - `feat: 〇〇機能を追加`
  - `fix: 〇〇のバグを修正`
  - `refactor: 〇〇を整理`
  - `docs: ドキュメントを更新`
  - `chore: 設定・依存の更新`
- コミット・push は**利用者が頼んだときだけ**行う。勝手に push しない。
- `main` で直接作業しそうなときは、先にブランチを切る。

---

## 禁止・注意事項

- API キーやパスワードなどの**秘密情報をコミットしない**（`.env` は `.gitignore` 済み）。
- `node_modules` や生成物（`dist`）はコミットしない。
- 確認なしで破壊的操作（`rm -rf`、`git reset --hard`、強制 push）をしない。
- 仕様が不明なまま大量のコードを生成しない。まず小さく作って確認する。
